# v1.4.0 Further Development Scope

Based on the implemented features in the `v1.4.0` Web Components slice (including the `ta-quest-log`, `ta-faction-board`, `ta-relationship-web`, and `ta-route-planner`), the following areas have been identified as spaces for further development and system deepening.

## 1. Deepening Faction and Relationship Systems

The visual foundations for factions and relationships are now in place via the `ta-faction-board` and `ta-relationship-web` widgets. Future iterations should focus on mechanically integrating these standings into the core game loop:

- **Dialogue Integration:** Faction standings could dynamically unlock or restrict specific dialogue choices with NPCs belonging to those factions.
- **Economic Impact:** Vendor prices and trading availability could scale dynamically based on the player's relationship with the controlling faction.
- **Emergent Encounters:** Factions with hostile standings might trigger dynamic ambushes or restrict access to certain map zones, while allied factions could provide support or safe havens.
- **Dynamic AI NPC Behaviors:** Hook faction standings into the AI NPC generation prompts, allowing NPCs to dynamically react to the player's reputation (e.g., refusing to speak, offering bribes, or calling for guards).

## 2. Advanced Map Routing and Travel Mechanics

The `ta-route-planner` successfully calculates shortest known unlocked routes, travel time, and supply costs. This system can be expanded to make travel more dynamic and risky:

- **Dynamic Hazards:** Routes could have fluctuating hazard levels based on world events, weather, or faction wars, forcing the player to choose between a faster, dangerous route or a slower, safer one.
- **Randomized Encounters:** Travelling across multiple zones could introduce procedural encounter checks (e.g., ambushes, merchants, environmental obstacles) that interrupt the planned route.
- **Supply Management Escalation:** Introduce more complex supply requirements (e.g., specific gear or vehicles) for traversing hazardous terrain (like deep space or toxic swamps) identified during route planning.
- **Fog of War & Scouting:** Allow players to use skills or spend resources to reveal details about adjacent unknown nodes on the `ta-world-atlas` without travelling there.

## 3. Quest and Lore Expansion

With the robust `ta-quest-log` and `ta-clue-board` systems implemented:

- **Complex Quest Triggers:** Quests could trigger based on specific thresholds in faction standing or the accumulation of particular clues across multiple regions.
- **Time-Sensitive Quests:** Introduce deadlines or shifting priorities for quests, utilizing the existing `ta-ticker` system to create urgency.
- **Lore Integration:** Expand the `ta-codex` to automatically cross-reference discovered clues, creating a "detective board" experience where players can manually link clues to form deductions and unlock new quest objectives.

## 4. WebGL Dice & Visual Polish

The deterministic WebGL dice roller provides a great foundation for visual flair:

- **Custom Dice Skins:** Allow visual styles (e.g., neon, weathered, brutalist) to dictate the material, color, and particle effects of the 3D dice.
- **Critical Hit/Fail VFX:** Add screen-shake or localized particle explosions when rolling a natural 20 or a critical failure (e.g., percentile 00).
- **Environmental Lighting:** Use the current scene's atmosphere attributes to tint the lighting on the WebGL canvas, grounding the dice in the game's current environment.

## 5. Authoring & GM Tooling

To support the creation of these complex systems:

- **Live Preview CLI:** A command (e.g., `tag dev` or `tag preview`) that watches file changes and live-reloads the scene or widget being worked on, heavily utilizing the custom elements.
- **Balance Validation:** CLI tools to analyze a scenario and flag potential balance issues (e.g., "Quest X requires a stat of 18, but is accessible at Level 1").
- **Timeline Chronicle:** A visual "end of game" or "recap" widget that plots major decisions, faction changes, and quest completions on a chronological timeline, utilizing SVG graphing similar to `ta-relationship-web`.

## 6. Audio & Accessibility Improvements

Building on the `SoundscapeEngine` and existing accessibility features:

- **Spatial Audio:** Tie audio panning to map locations or incoming hazards.
- **TTS Voice Profiles:** Allow different NPCs to utilize distinct TTS pitch/rate profiles based on their data attributes.
- **High-Contrast Themes:** Introduce a dedicated high-contrast accessibility CSS theme that strips out complex background textures in favor of maximum readability for all Web Components.

## 7. Quality & Testing Enhancements

To ensure the robustness of the CLI and generated Web Components, the following static analysis and automated tests should be introduced into the `cli/tests/quality/` suite:

- **Architectural Import Boundaries (`imports.spec.ts`):** Implemented in `cli/tests/quality/imports.spec.ts`; files inside `cli/render/templates/` and `cli/render/lib/` must not import Node-specific modules (like `node:fs` or `node:path`) or reference host runtime globals such as `Bun` and `process`.
- **TDD Parity Validator (`tdd-parity.spec.ts`):** A formal test to programmatically crawl the `cli/` directory, read `.tddignore`, and assert that every production `.ts` file has a corresponding `.spec.ts` file, providing continuous visibility during CI/CD.
- **CSS Variable Consistency (`css-vars.spec.ts`):** Implemented in `cli/tests/quality/css-vars.spec.ts`; generated CSS plus render/style sources must either resolve custom property usages to declarations or provide explicit `var(--token, fallback)` values.
- **Console Logger Leakage (`no-console.spec.ts`):** Scan all files outside the main `cli/commands/` and the entry point to ensure that `console.log`, `console.warn`, or `console.error` are not inadvertently left in library or rendering functions, protecting the CLI's standard output streams.
- **Metadata Sync Check (`metadata-sync.spec.ts`):** Programmatically scan the `cli/render/templates/` directory to verify that every template file exports a render function, and that each function perfectly corresponds to a registered widget in `cli/metadata.ts`.

## 8. Browser/Web Component Runtime Audit

The Bun runtime harness catches hydration and event logic, but it does not exercise a real browser layout engine. A dedicated browser audit should cover:

- **Playwright Smoke Matrix:** Render every custom element in Chromium at desktop and mobile widths, then assert the element upgrades, attaches shadow DOM where expected, and exposes visible, non-empty content.
- **Interaction Coverage:** Exercise primary controls for `ta-scene`, `ta-quest-log`, `ta-faction-board`, `ta-relationship-web`, `ta-route-planner`, and dice widgets with actual click and keyboard events.
- **Canvas and SVG Rendering Checks:** Capture screenshots or pixel probes for WebGL dice, SVG graphs, and icon sprites to catch blank canvases, missing symbols, or bad viewBox scaling.
- **Fixture Discipline:** Use deterministic state fixtures generated through the CLI rather than hand-authored HTML so the audit validates the real render pipeline.

## 9. Security Dependency and Surface Audit

Static code checks should be paired with a dependency and package-surface review before release:

- **Dependency Audit:** Run the package manager audit path for production and dev dependencies, and record any accepted risks with dates and package versions.
- **Package Surface Review:** Verify that `bin`, exports, generated assets, and checked-in bundles expose only intended files and do not publish test fixtures, temp output, or local machine paths.
- **Path and File IO Review:** Re-run the safe-path tests against representative home, temp, `/mnt`, missing-file, symlink, and traversal cases.
- **HTML/SVG Sanitisation Review:** Keep SVG sprite generation and HTML data-attribute escaping under direct test so hostile SVG/script input cannot bypass the renderer.

## 10. Performance Budgets

The generated Web Component assets should have explicit size and runtime budgets so regressions are visible:

- **Generated Asset Budgets:** Implemented in `cli/tests/quality/asset-budgets.spec.ts` with reviewed limits in `cli/tests/quality/asset-budgets.json`; enforces upper bounds for `ta-components.js`, per-style CSS, total CDN asset size, JS total, CSS total, and icon sprite size.
- **Render Output Budgets:** Track HTML output size for representative scene, quest, map, faction, and relationship widgets.
- **Hydration Cost Budgets:** Measure real-browser custom-element upgrade time for heavier widgets and fail only on clear regressions, not small noise.
- **Fixture-Based Trend Data:** Store baseline fixture names and budget numbers in a small config file so changes are intentional and reviewable.

## 11. Accessibility Checks

Accessibility should be tested at the custom-element boundary rather than only by visual inspection:

- **Keyboard Navigation:** Validate focus order and keyboard activation for buttons, tabs, route controls, quest controls, and reveal panels.
- **Accessible Names:** Check that icon-only controls, canvas widgets, SVG graphs, and custom elements expose useful names or labelled fallbacks.
- **Contrast and Reduced Motion:** Run automated contrast checks for high-use states and verify reduced-motion paths disable nonessential animation.
- **Screen Reader Structure:** Assert headings, landmark-like regions, status updates, and hidden content states are coherent inside rendered custom elements.
